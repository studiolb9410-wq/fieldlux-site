Judgment Engine

Auto-verdict on every install.

Build the scene, choose the lens, aim at the target. The analyzer returns a verdict in one of three states — the states themselves are stable; the underlying thresholds are still being tuned against field data during closed beta.

READY · ships within mount/drift tolerance
AT_RISK · passes in best-case conditions, margin is thin
NOT_RECOMMENDED · at least one core metric is likely to fall short

Each verdict is supported by the install checks that matter most on site — pixel density, bright-enough coverage, target coverage, contrast margin, spill, blend hotspot. When a metric falls short, you see which one and what to adjust. The full check list is being expanded during beta; thresholds and reasoning are documented in the methodology section.

Stress Analysis

Know every failure mode before
the flightcase leaves the shop.

S0 through S10 simulate the alignment drift projectors see on real sites — rigging settle, mount movement, thermal shift. The current tolerance profile is still being tuned as we collect more install data:

Seven review profiles shift the acceptance bar depending on the stage — a quick setup check uses a different profile than a live-show review or a long-term archival review. The resilience score is available before any crew arrives on site, and the profile set is being expanded during beta.

Prescription Engine

Concrete actions — not just warnings.

When brightness fails, we don't say "low lux". We say:

Move the projectors 40 cm closer as a group
    — or —
step up to a higher-lumen unit
    — or —
swap to a shorter-throw lens.

Distance and lumen suggestions are derived from documented physics — inverse-square, lens-throw, blend-footprint geometry. Lens and gear swap suggestions fall back to documented heuristics, shown in full in the prescription card so the reasoning is auditable.

When two suggestions conflict — "closer for brightness" vs "back for coverage" — the suggestion that resolves the core failing metric takes priority, and the reason is shown alongside the suggestion. The conflict-resolution model is being refined during closed beta.

Multi-target Analysis

Analyze what you actually project onto.

Flat surfaces, room walls, imported building meshes, site photos — the same judgment engine runs on all four. Bring your model in as glTF, FBX, OBJ, or Collada. Lens-warped projection lands on the real mesh; when a ray misses, it falls back to a bounding-box centroid (that fallback's error budget is in the methodology section too).

Flat surface
Screen or scrim
Rectangular projection surface at a fixed throw. The baseline case — same judgment engine runs on all the rest.
Room walls
Interior space
Front, back, left, right, floor — five surfaces judged independently. Useful for immersive rooms and corner blends.

glTF · FBX · OBJ · Collada

Reliability

Results that stay consistent between sessions.

Each release is validated through an automated test suite covering the physics, the install-flow, and an engineering audit pass before it ships. Save → reload → reanalyze is targeted to stay within a tight lux window so a project review reads the same when you open it next week, or when a teammate opens it on their machine.

Calibration is pinned against real projector fixtures in the test suite, so regressions in perturbation or alignment math surface before a release goes out. During closed beta, the fixture set and thresholds continue to be refined as more install data comes in.
